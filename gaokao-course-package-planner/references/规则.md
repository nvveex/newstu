# 规则

## 一、硬约束

默认硬约束如下：

1. 每个学生需要覆盖自己的全部目标课程。
2. 每个学生在每个课位包中恰好上一门课。
3. 每个课程班只能落在一个课位包。
4. 同一教师同一课位包不能带两个班。
5. 课程班人数必须落在允许范围内。
6. 若用户要求 `3` 个包或 `6` 个包，按用户指定数量建模。

## 二、默认班额

如果用户没有额外说明，可以先按下面思路处理：
- 有明确上下限时，严格用用户上下限
- 没有明确上下限时，不要擅自假设，先问
- 如果用户已经明确“尽量均衡”，就用教师班级数量和总人数推导均衡目标

## 三、默认优化优先级

只有在用户没有重新指定时，才使用以下默认优先级：

### 1. 科目优先级

优先满足：
`物理 > 化学 > 地理 > 生物`

相对可调：
`历史、政治`

### 2. 分层优先级

在成绩分层模式下：
- 优先保留物理和化学的原始分层
- 优先保留物理和化学的 `A`
- `A` 尽量少进人
- `A` 的总人数尽量小

### 3. B/C 结构约束

如果存在 `A/B/C` 三层：
- `B` 班中不能让大量 `C` 层学生占多数
- `C` 班中也不应大量混入 `B` 层学生

常用表达方式：
- `B` 班里原始 `C` 层学生不超过一半
- `C` 班里原始 `B` 层学生不超过一半

### 4. 总调整原则

在满足可行性的前提下：
- 尽量少调整学生原始分层
- 能不调就不调
- 必须调时，优先让低优先级科目承担调整

## 四、异常处理规则

### 1. 教师表重复

如果用户确认只是重复录入：
- 按 `课程班 + 教师` 去重

如果用户没有确认：
- 先问，不要直接去重

### 2. 学生数据缺失

如果某个学生某科没有原始分层：
- 先列为异常
- 询问用户处理方式

如果用户明确要求忽略或移除：
- 从求解中移除，或保留为异常未满足
- 在结果里写明原因

### 3. 表间冲突

如果 `新科目组合` 与单科列冲突：
- 以用户确认为准
- 若用户未说明，先提问

## 五、解释规则

只要某个学生没有保留原始分层，就要能解释原因。  
常见原因包括：
- 若保留原分层，会与某个已固定课程班发生课位冲突
- 原分层班额已达上限
- 为满足教师同课位不冲突而调整
- 原始数据缺失或命名异常

## 六、结果汇报规则

汇报时优先说明：
1. 是否求得可行解
2. 一共多少个调整
3. 哪些高优先级科目是否被动到
4. 哪些学生或科目无法完全满足
5. 输出文件位置
