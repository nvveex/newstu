# 高考六选三课包规划

在用户已经提供学生选科数据、成绩分层数据或教师课程班数据时，使用这个技能完成数据核对、约束确认、求解和导出。

## 主流程

1. 先识别工作簿类型。  
如果只有 `选考-源数据` 和 `教师列表`，按“无分层模式”处理。  
如果同时有 `选考-源数据`、`合格考-源数据`、`教师列表`，按“成绩分层模式”处理。  
读取 [references/知识库.md](references/知识库.md) 了解表结构和命名规范。

2. 先做核验，再做求解。  
检查：
- 工作表名称是否符合预期
- 学生行数是否一致
- `新科目组合` 与各科列是否一致
- 教师表是否有重复课程班记录
- 是否存在缺失分层、错列、异常命名、重复学生、空行

3. 先把不确定项一次性列给用户确认。  
如果发现异常学生、字段缺失、课程班命名冲突或规则口径不清，先汇总成一份简短问题清单让用户确认。  
不要在数据有歧义时直接盲猜继续求解。

4. 统一做标准化。  
统一全角/半角数字字母。  
统一 `合考 -> 合格考`。  
统一是否带 `班` 后缀。  
按 `课程班 + 教师` 去重时，先确认这些重复行是否只是重复录入。

5. 按用户规则建模。  
硬约束至少包括：
- 每个学生必须在自己的全部目标课程中各分到一个课程班
- 每个学生在每个课位包里只能上一门课
- 每个课程班只能落在一个课位包
- 同一教师同一课位包内不能带两个班
- 班额必须满足当前任务要求

6. 再处理优化目标。  
如果用户给了优先级，严格按用户优先级。  
如果用户没有给优先级，读取 [references/规则.md](references/规则.md) 使用默认规则。  
需要解释清楚“为什么某些学生没能保留原分层”。

7. 求解后必须复核。  
至少检查：
- 每个课位包的课程班数量是否正确
- 每个课位包是否完整覆盖全体学生
- 是否还有教师冲突
- 是否还有学生同课位冲突
- 未满足名单是否真的只有无法兼容的情况

8. 输出两类结果。  
详细版：适合校内排课、复盘和二次调整。  
教师简洁版：适合直接发给老师。  
具体表结构见 [references/工作流.md](references/工作流.md)。

## 默认执行规则

1. 用户规则高于默认规则。
2. 只要存在数据歧义，就先提问确认。
3. 如果只是重复录入且用户已确认可去重，就按去重后处理。
4. 如果是少量异常学生且用户明确要求忽略，要把这类学生记录进“未满足原因”或“说明”。
5. 输出时优先给出可执行结果，其次再解释求解过程。
